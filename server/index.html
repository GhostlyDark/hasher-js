What does it all mean?

Hasher-js gives you a lot of info. The most important things to look at are the file hashes and database matches. Skip ahead if you want to learn what all this "hashing" business is about.

  • A database match is a step in the right direction but it does not necessarily mean you have exactly the right file. Always check the hashes.
  • If you have the right ROM hash but the wrong file hash, here's what it might mean:
    • NES - Most likely, there is "garbage" in the header (extra data that shouldn't be there). This won't typically cause a problem, but it can be removed using a header cleaner program. However, it's also possible that the header contains incorrect data which might cause a problem with certain emulators when using a patched ROM.
    • SNES - Generally caused by external headers. There are tools that can add or remove external headers (such as SNESStuff). If your ROM hash matches and the ROM meets the header requirements of the patch, you should be safe, even if the file hash doesn't match.
    • Genesis - Genesis ROMs typically come in two different formats: SMD and BIN. A tool such as SBWin can convert between these formats.
    • N64 - ROMs may be big-endian, byte-swapped or little-endian. A tool such as Tool 64 can convert between these different formats.

What's all this "hashing" business, now?

The filename doesn't tell you everything you need to know. You can run into bad dumps, overdumps, corrupt files, mislabeled files, and more.

Hashing is like a form of fingerprinting for data. Checking the hash of a file helps you determing whether you have exactly what you're looking for. Hasher-js provides a hash for both the whole file and the ROM image contained inside the file, plus some additional data, to help you figure out whether you have the right thing in the right format.
